Rewrite manifest.json to include all Phase 2 additions while preserving the existing content_scripts block. The complete manifest must be:

```json
{
  "manifest_version": 3,
  "name": "Hebrew RTL",
  "version": "0.2.0",
  "description": "Hebrew text direction correction at element level",
  "permissions": [
    "storage",
    "activeTab",
    "tabs",
    "scripting"
  ],
  "background": {
    "service_worker": "background.js",
    "type": "module"
  },
  "content_scripts": [
    {
      "matches": ["<all_urls>"],
      "js": ["content.js"],
      "run_at": "document_idle"
    }
  ],
  "action": {
    "default_title": "Hebrew RTL"
  },
  "commands": {
    "toggle-rtl": {
      "suggested_key": {
        "default": "Ctrl+Shift+H",
        "mac": "MacCtrl+Shift+H"
      },
      "description": "Toggle Hebrew RTL correction for current domain"
    }
  }
}
```

Key changes from Phase 1:
- `version`: "0.1.0" -> "0.2.0"
- Added `permissions`: ["storage", "activeTab", "tabs", "scripting"]
- Added `background`: service_worker pointing to background.js with type: module
- Added `action`: default_title for badge support
- Added `commands`: toggle-rtl with suggested_key (Ctrl+Shift+H / MacCtrl+Shift+H)
- Removed `"type": "module"` from content_scripts (research flags this as NOT officially supported for content scripts; content.js will need to avoid ES module imports or use a different pattern -- see 02-04 for content.js wiring)

NOTE: The `"type": "module"` removal from content_scripts is intentional. Research (02-RESEARCH.md line ~150) documents that Chrome's official manifest docs do NOT list `type` as a valid content_scripts field. The background service worker DOES support `"type": "module"` (confirmed). For content.js, plan 02-04 will handle the import resolution by either confirming module support works or inlining the dependencies.
